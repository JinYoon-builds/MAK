# MAK UI/UX 개발 방향 — 두 축

MAK 키오스크의 목표는 **"어르신이 그냥 와서 말하면, 알아서 깔끔하게 발권까지 이어지는"** 경험이다.
이걸 UI/UX 관점에서 분해하면 **딱 두 축**으로 나뉜다. 하나는 사람 → 기계로 들어오는 쪽(인풋),
하나는 기계 → 사람으로 나가는 쪽(반응 화면). 두 축은 독립적으로 좋아질 수 있고, 지금은
**인풋 축이 앞서 있고 아웃풋 축이 뒤처져 있다.**

```
        (축 1) 인풋: 언제 말할지 고민 안 해도 발권이 진행되는가
  사람  ───────────────────────────────────────────────▶  기계
        ◀───────────────────────────────────────────────
        (축 2) 아웃풋: 되돌려주는 화면이 인터랙티브한가
```

---

## 축 1 — 인풋: "언제 말해야 하는지 생각하지 않아도 되는가"

**질문:** 어르신이 화면 앞에 서서, *말할 타이밍을 고민하지 않고* 그냥 말하면 그게 깔끔하게
잡혀서 발권이 진행되는가?

**성공 기준**
- 안내가 끝나길 기다릴 필요가 없다(아무 때나 말 걸 수 있다).
- 첫 음절부터 끝까지 유실 없이 잡힌다.
- 느리게·중간에 쉬며 말해도 중간에 잘리지 않는다.
- 옆 사람 목소리·기계 자기 안내음은 입력으로 새지 않는다.
- 마스크·역광·고개 돌림 같은 악조건에서도 **진짜 발화를 버리지 않는다**(애매하면 통과 → 확인).

### 구현됨 (코드에 있음)

| 기능 | 위치 / 상수 | 한 줄 설명 |
| --- | --- | --- |
| 존재 게이트 | `detectCloseFace`, `closeRatioPercent`, enter/exit 디바운스 | 충분히 가까운 얼굴이 보일 때만 청취 시작 |
| 입술 모션 신호 | `src/lipMotion.ts` `LipMotionTracker` | 블렌드셰이프 벌어짐 **진폭**(자세에 강건) |
| 프레임 붕괴 방지 | `LIP_MOTION_MIN_SAMPLES`, gap 리셋 | 저프레임에서도 진폭이 0으로 안 무너짐(버그 #2 수정) |
| confidence 분기 폐기 | `handleVoiceTurn`, `LIP_STILL_FLOOR`, `faceObserved` | 얼굴 관찰 + **완전 정지**일 때만 폐기(버그 #1 수정) |
| barge-in | `maybeBargeIn`, `BARGE_IN_MIN_TICKS` | 안내 중 입술 움직이면 즉시 청취(카메라 판정→에코 무관) |
| 캡처 중 존재 유예 | `CAPTURE_PRESENCE_GRACE_MS` | 말하다 고개 숙여 얼굴 사라져도 턴 유지 |
| 에코 제거 | 실시간·폴백 `getUserMedia` AEC | 자기 TTS 되먹임 억제 |
| STT | 실시간(OpenAI, 서버 VAD) + 폴백(녹음→파일 STT) | 실시간 우선, 실패 시 폴백 |

### 테스트해야 할 것 (실측 필요 — 상수 대부분 미검증)

- [ ] **barge-in이 실제로 걸리는가** — 벤치에서 입 움직일 때 `lip` 점수가 임계값(0.7)을 넘는지 먼저 확인. (지금 "안 넘어감" 증상의 원인 확정 필요)
- [ ] **진짜 발화를 잘못 버리지 않는가** — 마스크·역광·작은 목소리에서 폐기 오발 여부
- [ ] **바이스탠더 차단이 되는가** — 앞사람 침묵 + 옆사람 말할 때 실제로 걸러지는지
- [ ] **엔드포인팅 타이밍** — 느리게·중간에 쉬며 말할 때 서버 VAD가 중간에 자르는지
- [ ] **존재 감지 강건성** — 조명/각도/거리/안경/키 차이에서 안정적인지
- [ ] **AEC 실효** — 실제 키오스크 **외부 스피커** 환경에서 잔향이 남는지(브라우저 오디오 그래프 밖)
- [ ] **프레임레이트** — 목표 키오스크 하드웨어에서 블렌드셰이프 추론이 150ms 틱을 따라오는지
- [ ] **임계값 보정** — `lipMotionThreshold`·`closeRatioPercent`·`silenceMs`를 실제 사용자로 튜닝

### 더 발전시켜야 할 것

- **프리롤 버퍼 (Phase 2, 최우선)** — barge-in 직후 마이크 세팅 사이(0.5~1.5s)에 유실되는 첫 음절을
  상시 오디오 링버퍼로 물었다 붙여 살림. **현재 알려진 한계이자 "안 넘어감"의 유력 원인.**
- **입술 보조 적응형 엔드포인팅 (Phase 2)** — 고정 침묵 타이머 대신, 소리가 멈춰도 입이 움직이면 대기 연장 / 입도 멎으면 조기 종료
- **동반자 발화 구분 강화 (Phase 2)** — 지금은 근접 얼굴 1개 기준. 근접 얼굴 입술과 오디오 onset의 **동시성**으로 좁히기
- **존재 감지 민감도 완화** — 조명/각도에 덜 민감하게(멀티 프레임 평균, 신뢰도 활용)
- **저장 보정값 마이그레이션(정리)** — 점수 스케일이 바뀌었는데 `mak.vadCalibration.v1` 키는 그대로라, 예전에 저장한 `lipMotionThreshold`가 새 스케일에 잘못 적용될 수 있음 → 키 버전업 또는 리셋 필요(리뷰 지적 미반영분)

> 이 축은 이미 궤도에 올라 있다. "언제 말할 수 있냐"는 Phase 1에서 크게 풀렸고, "그 말을 온전히
> 잡냐"(첫 음절·말 끝·바이스탠더)가 Phase 2의 몫이다. **대부분의 튜닝 상수가 아직 실측 미검증**이라는 점을 유념.

## 축 2 — 아웃풋: "되돌려주는 화면이 인터랙티브한가"

**질문:** 인풋에 대한 반응으로 어르신에게 정보를 전달하는 화면이, 상황에 맞게 살아 움직이며
사용자를 이끄는가? 아니면 정해진 순서를 그냥 재생만 하는가?

**현재 상태 (뒤처진 축)**
- 화면은 **고정된 유한상태기계(FSM)** 다: `ScreenId` 12종을 `show()`로 전환.
- LLM은 "다음에 어떤 preset 화면"인지 **고를** 뿐, 화면 자체를 **설계/수리하지 않는다.**
- 즉 지금은 *"입력·판단은 에이전트적, 화면은 결정론적 스크립트"* 상태.

**방향 — mixed-initiative(주도권을 주고받기)**
"에이전트가 정해진 화면 중 고르기 → 에이전트가 상호작용을 설계·수리하기"로 옮긴다.
- **아무 순서로 던져도 받는다.** "내일 아침 부산 두 장"을 한 번에 → 남은 슬롯만 되물음.
  (LLM은 이미 multi-slot 파싱하므로 **화면 흐름만 슬롯 기반으로** 바꾸면 됨.)
- **능동적 명확화(self-repair).** 모호/오인식을 스스로 감지해 되물음("부산이었을까요, 광주였을까요?").
- **상황 적응.** 재확인이 잦은 사용자면 확인 강도를 높이고, 선택지를 줄이고, 글씨를 키우는 식으로
  화면 요소를 상황에 맞춰 조절(적응형 UI). 극단은 generative UI지만 이 도메인엔 과함(아래 경계 참고).
- **투명성·되돌리기.** 왜 그렇게 했는지 말하고, 언제든 취소/정정 가능.

**세부 과제(초안)**
- FSM의 화면 순서를 **슬롯 상태 기반 라우팅**으로 재구성(고정 dest→confirm→time→summary 사슬 완화)
- dialog turn 결과에 **명확화 액션**을 1급 시민으로 추가(현재 `retry_listening`보다 세분화)
- 화면 컴포넌트를 **상태에 반응하는 단위**로(정보 밀도·선택지 수·재확인 강도를 파라미터화)

## 두 축을 관통하는 원칙

1. **틀려도 회복되는 UX.** 인풋 엔드포인팅도, 아웃풋 라우팅도 가끔 틀린다. 완벽을 기대하지 말고
   **항상 정정 가능한 확인 단계**로 감싼다. 거짓 무시가 거짓 통과보다 비싸다.
2. **개입 비용이 크면 확신이 높을 때만 주도한다** (Horvitz의 mixed-initiative 원칙). 어르신 +
   결제 도메인은 틀렸을 때 당황 비용이 크므로, 애매하면 사용자에게 주도권을 넘긴다.
3. **예측가능성 > 자율성 (이 도메인 한정).** 화면 골격은 읽히고 예측 가능하게 둔다. 매번 달라지는
   완전 생성형 UI는 오히려 혼란·불신을 준다. 스윗스팟은 **강한 mixed-initiative + 강한 회복성**이지,
   완전 자율 생성형 UI가 아니다.

## 우선순위

- **축 1 Phase 2**(프리롤·적응형 엔드포인팅)를 먼저 마무리해 "인풋이 깔끔하다"를 확정한다.
- 그 다음 **축 2**의 슬롯 기반 mixed-initiative 라우팅 + 능동적 명확화로 넘어간다.
- 두 축 모두 위 3원칙(회복성·확신기반 주도·예측가능성)을 지킨다.

> 참고: "Agentic UI"는 아직 합의된 학술 용어가 아니라 mixed-initiative 계보의 재포장에 가깝다.
> 용어에 끌려가지 말고 **"인풋이 깔끔한가 / 아웃풋이 상황에 반응하는가"** 라는 두 질문으로 판단한다.
